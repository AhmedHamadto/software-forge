# Infrastructure as Code (Kief Morris) — Teaching Content

## IaC Patterns

### Deep (first exposure)

Before Infrastructure as Code, provisioning a server meant logging into a console, clicking through menus, and hoping the person who set up the production environment remembered all the same steps they used in staging. Morris opens Chapters 1 through 4 with the fundamental problem: manually managed infrastructure is unreproducible, undocumented, and fragile. IaC treats infrastructure definitions as source code — version-controlled, reviewed, tested, and applied through automated pipelines. The result is that your infrastructure becomes as auditable and repeatable as your application code.

The first design decision is declarative versus imperative. Declarative tools (Terraform, CloudFormation, Pulumi with declarative mode) describe the desired end state — "there should be three EC2 instances behind a load balancer" — and the tool figures out what changes to make. Imperative tools (scripts, Ansible playbooks, CDK with procedural logic) describe the steps to get there — "launch an instance, then configure it, then register it with the load balancer." Morris advocates for declarative approaches because they are idempotent: running the same definition twice produces the same result. Imperative scripts, if not carefully written, can create duplicate resources or fail on re-execution. However, declarative tools introduce state management complexity. Terraform, for example, maintains a state file that maps your definitions to real resources. If that state file gets corrupted, lost, or out of sync with reality (someone made a manual change in the console), the tool's view of the world diverges from the actual world, and the next apply can destroy resources or fail in confusing ways.

Module boundaries and environment promotion are where IaC becomes an architecture problem. Morris describes how infrastructure definitions should be decomposed into modules — networking, compute, database, monitoring — with clear interfaces between them. A change to the networking module should not require changes to the compute module. Environments (dev, staging, production) should use the same modules with different parameter values, not separate copies of the code. Promotion means a change flows dev to staging to production, validated at each stage, just like application code. The plan-before-apply pattern is critical: every change should produce a preview of what will be created, modified, or destroyed, and a human (or an automated policy check) should review that plan before it executes. Blast radius management means structuring your state and modules so that a single apply cannot accidentally destroy your entire production environment. Separate state files per environment, per service, or per layer ensure that a mistake in one area cannot cascade.

Without these patterns, IaC becomes a different kind of fragility. Teams who dump all their infrastructure into a single Terraform state file with no module boundaries discover that a typo in a DNS record definition triggers a plan that wants to recreate their database. Teams who do not use plan-before-apply discover that their CI pipeline destroyed a production load balancer because someone merged a formatting change that the tool interpreted as a resource replacement. Morris's patterns are not about choosing the right tool — they are about structuring your infrastructure code so that changes are safe, reviewable, and incremental.

### Refresher (subsequent exposure)

IaC treats infrastructure definitions as version-controlled, reviewed, and tested source code. Use declarative definitions for idempotency, manage state carefully, decompose into modules with clear boundaries, promote changes through environments (dev to staging to prod), always plan before applying, and limit blast radius by separating state files per environment and service.

### Socratic Questions

- Your team stores all Terraform state for production in a single state file. A developer modifies a minor tag on an S3 bucket, runs `terraform apply`, and the plan shows it wants to recreate the production RDS database. What structural change to your IaC would prevent this kind of blast radius, and how would you decompose the state?
- A new engineer joins and notices that the staging environment's infrastructure code is a copy-paste of production with different values hardcoded throughout. They want to add a new monitoring resource, which means making the same change in two places. What IaC pattern would eliminate this duplication, and what risks does the current approach create?
- Your CI pipeline runs `terraform apply` automatically on merge to main, without a plan review step. A developer accidentally removes a security group rule in their PR. What happens, and what workflow change would catch this before infrastructure is modified?
